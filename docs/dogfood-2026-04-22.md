# Dogfood pass — 2026-04-22

First end-to-end run of Efterlev against a real Terraform codebase —
the govnotes-demo synthetic boundary — rather than the hand-crafted
fixture the E2E smoke uses. govnotes-demo carries a `DELIBERATE_GAPS.md`
with 12 ground-truth compliance gaps, so this pass doubles as a
coverage-accuracy measurement against a known baseline.

## Run parameters

- **Target:** `govnotes-demo/infra/terraform/` only (staging env is
  explicitly out of scope per `DELIBERATE_GAPS.md`). 13 `.tf` files,
  84 parsed resources.
- **Efterlev branch:** `main` at `a68ec7d` (12 detectors, post-Phase-6-lite).
- **Pipeline:** `init → scan → agent gap → agent document → agent remediate --ksi KSI-SVC-VRI` via `uv run efterlev …`.
- **Artifacts:** under `.dogfood/20260423T002344Z/` (gitignored).

## Findings ranked by leverage

### 1. Terraform module `for_each` blind spot (CRITICAL)

`govnotes` declares 5 buckets through a storage module with
`for_each = var.buckets` over a map. Our HCL parser sees ONE
`aws_s3_bucket "this"` resource declaration (inside the module) and
cannot resolve it into per-key buckets without executing Terraform.
Concretely, `scan_terraform` found 2 `aws_s3_bucket` resources total
when the applied state contains 5+. Ground-truth gaps #1 and #2 (both
on the `user_uploads` map entry) are invisible as a direct
consequence, and the Remediation Agent couldn't surface the showcase
gap for the same reason.

**Fix path:** Terraform Plan JSON support is already named in the v1
locked plan (`docs/dual_horizon_plan.md` §3.1 Month 1: "Source
expansion: Terraform Plan JSON support (scans resolved plans
including computed values)"). Pulling that forward is the single
highest-leverage follow-up after this dogfood — module-heavy
codebases are the common case, not the edge case, and we're
underreporting by ~60% on buckets in this target.

### 2. `jsonencode(data.aws_iam_policy_document…)` limits MFA detection (HIGH)

All 7 IAM policies in govnotes declare their policy body via
`jsonencode(data.aws_iam_policy_document…)` rather than heredoc
literal JSON. Our `mfa_required_on_iam_policies` detector honestly
emits `mfa_required=unparseable` with a clear gap string — the
docstring anticipates this — but it means we cannot detect MFA
enforcement at all on any of the real-world policies. Ground-truth
gaps #7 and #11 (and the "present-correctly" control for platform
admin) all land in the unparseable bucket. The Gap Agent honestly
surfaces this in its KSI-IAM-MFA rationale, but the customer value
of the detector collapses on this pattern.

**Fix paths, ranked:**
1. **Parse `data "aws_iam_policy_document"` blocks statically.** HCL
   declares the statement structure inline (Effect, Action, Resource,
   Condition) — Terraform evaluates it to JSON, but the static
   structure is observable from the HCL AST. Walk the
   `aws_iam_policy_document` data blocks, find those referenced by
   `jsonencode(data.aws_iam_policy_document.X.json)` in `aws_iam_policy.policy`,
   and check their `condition` blocks for `aws:MultiFactorAuthPresent`.
   Moderate effort, big payoff.
2. **Terraform Plan JSON support.** Once plan-JSON is in, this
   collapses — the resolved policy body is a plain JSON string the
   existing detector already handles.
3. **Accept the limitation and cover via Manifest.** Customer
   attests: "our IAM policies require MFA per our policy review
   process." Weakens the Evidence-Manifest distinction we've been
   deliberate about.

Option 1 is the right v1 investment; option 2 subsumes it in v1.5+.

### 4. Detector enhancements (MEDIUM)

- **`tls_on_lb_listeners`** sees HTTP listeners on port 80 but does
  not check whether the `default_action` is a `redirect` to HTTPS
  (the mitigating control for having port 80 open at all). Ground
  truth #5 would upgrade from "HTTP listener detected" to "HTTP
  listener without redirect" with this enhancement.
- **`cloudtrail_audit_logging`** checks for presence of
  `event_selector` blocks but not whether any of them declare
  `data_resource` entries for S3 / Lambda. Ground truth #12's
  data-events half would be caught by adding a second evidence
  field like `has_data_event_selectors: bool`.

Both are small changes to existing detectors, not new folders.

### 5. `ksis=[]` detectors' evidence is invisible to KSI-level reasoning (LOW-MEDIUM)

The honest `ksis=[]` posture (5 of 12 detectors) means their
evidence lands in the "unmapped" bucket in the Gap Agent prompt.
The Gap Agent can reach into that bucket but doesn't by default —
the KSI prompt leads with mapped evidence. Concretely, KSI-SVC-VRI
was classified citing only `fips_ssl_policies` evidence (which has
an explicit KSI claim), not the S3 / RDS / KMS evidence which all
touch "cryptographic integrity of resources" (KSI-SVC-VRI's stated
outcome).

**This is not wrong** — the discipline in DECISIONS 2026-04-21
design call #1 is deliberately conservative, and per-control-to-KSI
expansion is exactly the kind of stretching we explicitly chose not
to do. But when a future FRMR release adds SC-28 / SC-12 to a KSI's
`controls` array, the existing detectors should pick up the mapping
without a code change. Worth verifying the Gap prompt handles that
transition when it happens; not blocking now.

### 6. UX: scan output "Detector record IDs" list is noisy (LOW)

The final section of `efterlev scan` lists one line per evidence
record in the form `<sha256>  <resource_name>`. On govnotes, 11 of
31 records have a `resource_name` that appears on another record
too (e.g., `cloudtrail` = the trail, the bucket, the SSE resource,
and the backup retention record). A reader can't tell which detector
produced which without walking the provenance graph. Small
improvement: include the short detector id on each line.

## What this does NOT measure

- **Real FedRAMP-Moderate coverage ceiling.** We've covered 12
  detectors; the target is 30. Even at 30, scanner-only coverage
  tops out around the ~20% line per `docs/dual_horizon_plan.md`
  §3.1 — the remaining 60% is procedural (Evidence Manifests).
  govnotes-demo is synthetic SaaS infra; a real customer's posture
  depends heavily on whether they've authored matching manifests.
- **Plan-vs-code semantic accuracy.** We're reading `.tf` source,
  not `terraform plan -json`. A bucket that's provisioned via a
  local-source module with dynamic values (the `for_each` case) is
  invisible to us; one provisioned via plan JSON would be visible.
  Until Plan JSON lands, coverage on module-heavy codebases (most
  of them) is structurally understated.
- **Cost / token-count baseline.** First-run numbers only: Gap ~103s,
  Doc ~417s, Remediation ~21s. Real CI integration wants per-stage
  budget caps; out of scope here.
